## 4-Gates Diagram Re-rendering - IN PROGRESS (Evening)

### Overview
User requested re-rendering of all 4 diagrams following the comprehensive 4-gates approach from `MATH_DIAGRAM_RENDERING_INSTRUCTION.md`. Initial Stage 1 diagrams were exam-quality but did not fully implement the 4-gates methodology.

### The 4-Gates Approach (from MATH_DIAGRAM_RENDERING_INSTRUCTION.md)
- **Gate 0: Solve First** - Problem must be solved before choosing visual representation
- **Gate 1: Audience Constraint Check** - Verify methods are P5-P6 appropriate (no advanced notation, variables only when necessary)
- **Gate 2: Map Reasoning Chain** - Every solution step must have a visual anchor in the diagram
- **Gate 3: Plan Scale** - Choose appropriate pixels-per-unit scale for clarity

### Gap Analysis: Stage 1 vs. 4-Gates Requirements
**Stage 1 Diagrams (generated via generate_diagrams.py):**
- ✅ Correct values and measurements
- ✅ 300 DPI exam-quality output
- ✅ Clear labels and annotations
- ❌ **Missing:** Explicit reasoning chain visibility (Gate 2)
- ❌ **Missing:** Solution step annotations
- ❌ **Missing:** Visual anchors for each calculation step

**4-Gates Requirements:**
- Show volume calculations in tank diagram (Q7)
- Show relative speed calculation steps (Q9)
- Show daily sales calculations with reasoning (Q10)
- Show geometric property derivations (angle sum, linear pair) (Q13)

### Script Development Attempts

**generate_diagrams_v3.py:**
- Error: `TypeError: __main__.generate_q7_diagram.<locals>.to_iso() argument after * must be an iterable, not int`
- Root cause: Malformed edge list in 3D tank plotting
- Attempted custom `to_iso(x, y, z)` function failed

**generate_diagrams_v4.py:**
- Error: `TypeError: 'int' object is not iterable` at `ax.plot3D(*zip(*edge), ...)`
- Root cause: Edges list contained single coordinates instead of coordinate pairs
- Example: `([l, 0, h])` (single) vs `([l, w, 0], [0, w, 0])` (proper pair)

**Working Baseline:**
- `generate_diagrams.py` has correct edge structure with 12 proper 3D edge pairs
- Generates all 4 diagrams successfully at 300 DPI
- Missing only the 4-gates reasoning chain annotations

### Strategy Decision
Instead of debugging v3/v4 from scratch, created `generate_diagrams_4gates.py` that:
1. Builds on the working 3D plotting structure from `generate_diagrams.py`
2. Adds Gate 2 reasoning chain annotations to all 4 diagrams
3. Shows solution steps visually in each diagram
4. Maintains 300 DPI exam-quality output

## Stage 0 Finding 1 Applied — Fix 1: Diagram vs Question Data Separation (10:30 GMT+8)

### Overview
Stage 0 smoke test identified a critical issue: YAML files were mixing diagram data with question/solution data, causing renderer confusion about what to SHOW vs. what to HIDE.

### Issue Identified
- **Problem:** YAML structure had diagram data, question text, solutions, and answers all mixed together
- **Impact:** Renderer doesn't know which elements to display (diagram) vs. which to hide (answers, solutions)
- **Consequence:** Students could see answers or calculations before solving — violates solvability test

### Solution Applied
Created `YAML_SCHEMA_DIAGRAM_QUESTION_SEPARATION.md` with TWO CLEARLY SEPARATED sections:

#### 1. `diagram_data:` Section — VISIBLE ONLY
- **What to DRAW:** Shapes, labels, dimensions, annotations that appear visually on original diagram
- **NEVER INCLUDE:** Solutions, derived values, answers, calculations
- **Examples:**
  - Axes (for graphs)
  - Gridlines (if visible)
  - Data points (dots, values)
  - Vertex labels (A, B, C, D)
  - Color codes (if visible)

#### 2. `question:` / `solution:` / `answer_key:` Sections — NEVER RENDERED
- **What to KEEP HIDDEN:** Question text, solution steps, answer key, multiple-choice options
- **Renderer Responsibility:** Only read `diagram_data:` section, ignore other sections
- **Validation:** Student sees diagram + question text (separate), then solves independently
